# LedgerLens Feature Dictionary

Every feature produced by `detection/feature_engineering.py` is documented here.
Each entry states what the feature measures, how it is computed, its expected
value range (empirically derived from `data/synthetic_dataset.parquet` via
`scripts/compute_feature_ranges.py`), what high and low values indicate, and
whether the feature is **mutable** (can change as new trades arrive) or
**immutable** (fixed at the time of account creation).

> **Security note:** All range bounds and example values are derived exclusively
> from the synthetic dataset generated by
> `scripts/generate_synthetic_dataset.py`. No real Stellar mainnet wallet
> addresses or trade amounts appear anywhere in this document.

---

## How to read this document

| Column | Meaning |
|--------|---------|
| **Range** | `[min, max]` — the hard theoretical bounds; values outside this range trigger a `validate_feature_ranges` warning |
| **Empirical p1–p99** | 1st–99th percentile in the synthetic dataset; useful context for "typical" values |
| **High value** | What a high reading indicates about the wallet |
| **Low value** | What a low reading indicates about the wallet |
| **Mutable** | ✅ changes with new trades · ❌ fixed at account creation |

## 1 · Benford Features

Benford's Law states that in naturally occurring numerical datasets, leading
digit *d* appears with frequency log₁₀(1 + 1/*d*). Organic trading amounts
span orders of magnitude and conform to this distribution. Wash-trading bots
using fixed lot sizes or round numbers deviate systematically.

All three metric families are computed over five standard rolling windows:
**1 h · 4 h · 24 h · 168 h (7 d) · 720 h (30 d)**.

### 1.1 · `benford_chi_square_{h}h`

| Attribute | Value |
|-----------|-------|
| **Windows** | `h` ∈ {1, 4, 24, 168, 720} |
| **Formula** | χ² = Σ (observed_count_d − expected_count_d)² / expected_count_d for d = 1…9 |
| **Range** | [0, ∞) — unbounded above; values above ~15.5 are statistically significant at p = 0.05 (df = 8) |
| **Empirical p1–p99** | ≈ 0.1 – 99 (synthetic dataset) |
| **High value** | The digit distribution deviates significantly from Benford — strong wash-trade signal |
| **Low value** | Digit distribution conforms to Benford — consistent with organic trading |
| **Mutable** | ✅ |

Chi-square goodness-of-fit statistic comparing the observed leading-digit
frequency distribution of trade amounts over the trailing `h`-hour window
against the theoretical Benford distribution. Computed by
`detection.benford_engine.chi_square_statistic`.

---

### 1.2 · `benford_mad_{h}h`

| Attribute | Value |
|-----------|-------|
| **Windows** | `h` ∈ {1, 4, 24, 168, 720} |
| **Formula** | MAD = (1/9) Σ |observed_freq_d − expected_freq_d| for d = 1…9 |
| **Range** | [0.0, ~0.30] — theoretical max approaches 1/9 ≈ 0.111 per digit; practical ceiling ~0.08 |
| **Empirical p1–p99** | ≈ 0.0003 – 0.080 |
| **High value** | Non-conforming digit distribution; values > **0.015** trigger `benford_flag = true` (Nigrini 2012) |
| **Low value** | Close conformity with Benford's Law |
| **Mutable** | ✅ |

Mean Absolute Deviation between observed and expected digit frequencies over the
trailing `h`-hour window. The non-conformity threshold `MAD_NONCONFORMITY_THRESHOLD = 0.015`
is the standard Nigrini (2012) cutoff used throughout LedgerLens.

---

### 1.3 · `benford_z_max_{h}h`

| Attribute | Value |
|-----------|-------|
| **Windows** | `h` ∈ {1, 4, 24, 168, 720} |
| **Formula** | max over d = 1…9 of Z_d = (|obs_d − exp_d| − 1/(2n)) / sqrt(exp_d · (1−exp_d) / n) |
| **Range** | [0, ∞) — values above **1.96** indicate a single anomalous digit at p < 0.05 |
| **Empirical p1–p99** | ≈ 0.01 – 9.9 |
| **High value** | One specific digit is dramatically over- or under-represented (e.g. bots using 1 000-XLM lots inflate digit 1) |
| **Low value** | No individual digit is anomalous |
| **Mutable** | ✅ |

Maximum per-digit Z-score (with Nigrini continuity correction) across all nine
digits over the trailing `h`-hour window. Complements the chi-square and MAD
statistics by identifying *which* digit is anomalous.

---

### 1.4 · `benford_residual_chi_square_{h}h`

| Attribute | Value |
|-----------|-------|
| **Windows** | `h` ∈ {1, 4, 24, 168, 720} |
| **Formula** | Chi-square applied to the absolute STL residuals of the amount time-series |
| **Range** | [0, ∞) |
| **Empirical p1–p99** | ≈ 0.1 – 79 |
| **High value** | Residuals (unexplained variation after trend/seasonality removal) are non-Benford — adversarial amount crafting |
| **Low value** | Residual amounts conform to Benford |
| **Mutable** | ✅ |

Chi-square on STL residuals, hardened against seasonal amount patterns that
would inflate the raw chi-square. Set to `NaN` for windows with insufficient
data for STL decomposition.

---

### 1.5 · `benford_residual_mad_{h}h`

| Attribute | Value |
|-----------|-------|
| **Windows** | `h` ∈ {1, 4, 24, 168, 720} |
| **Formula** | MAD applied to the absolute STL residuals of the amount time-series |
| **Range** | [0.0, ~0.08] |
| **Empirical p1–p99** | ≈ 0.0001 – 0.069 |
| **High value** | Residuals are non-Benford |
| **Low value** | Residuals conform to Benford |
| **Mutable** | ✅ |

MAD on STL residuals. See `benford_residual_chi_square_{h}h` for context.

---

### 1.6 · `benford_ci_width`

| Attribute | Value |
|-----------|-------|
| **Formula** | Width of the bootstrap 95% CI for the chi-square statistic |
| **Range** | [0, ∞) · `NaN` when `BENFORD_CI_ENABLED=false` (default) |
| **Empirical p1–p99** | N/A (disabled by default) |
| **High value** | Uncertain chi-square estimate — likely fewer than ~30 trades in window |
| **Low value** | Tight CI — stable estimate |
| **Mutable** | ✅ |

Bootstrap confidence interval width for the chi-square statistic. Only populated
when `config.BENFORD_CI_ENABLED=true`. Used to flag wallets where the sample
size is too small for reliable Benford analysis.

## 2 · Trade Pattern Features

These features capture structural anomalies in *who* a wallet trades with and
*how* those trades are structured. They are independent of trade amounts and
therefore orthogonal to the Benford signal.

### 2.1 · `counterparty_concentration_ratio`

| Attribute | Value |
|-----------|-------|
| **Formula** | volume_with_top_counterparty / total_volume |
| **Range** | [0.0, 1.0] |
| **Empirical p1–p99** | ≈ 0.05 – 0.99 |
| **High value** | Near 1.0 — wallet routes almost all volume through a single counterparty; primary wash-trade signal |
| **Low value** | Near 0.0 — volume spread across many independent counterparties; consistent with a legitimate market maker |
| **Mutable** | ✅ |

Fraction of total traded volume transacted with the single most-used
counterparty. Computed by `compute_trade_pattern_features`.

---

### 2.2 · `round_trip_frequency`

| Attribute | Value |
|-----------|-------|
| **Formula** | trades_where_base_account == counter_account / total_trades |
| **Range** | [0.0, 1.0] |
| **Empirical p1–p99** | ≈ 0.01 – 0.92 |
| **High value** | Wallet is literally trading with itself — near-definitive wash-trade evidence |
| **Low value** | No self-trades detected |
| **Mutable** | ✅ |

Proportion of trades where `base_account == counter_account`. Any non-zero
value in a Stellar DEX context (where the protocol prevents true self-fills)
indicates that both sides of the trade share a funding source.

---

### 2.3 · `net_roundtrip_ratio`

| Attribute | Value |
|-----------|-------|
| **Formula** | Currently identical to `round_trip_frequency` |
| **Range** | [0.0, 1.0] |
| **Empirical p1–p99** | ≈ 0.001 – 0.91 |
| **High value** | High self-matching / round-trip activity |
| **Low value** | No round-trip activity |
| **Mutable** | ✅ |

Included as a distinct feature column to give ensemble models an independent
signal slot. Future implementations may use graph-based net flow to compute
genuine round-trip ratios across multi-hop paths.

---

### 2.4 · `self_matching_rate`

| Attribute | Value |
|-----------|-------|
| **Formula** | Currently identical to `round_trip_frequency` |
| **Range** | [0.0, 1.0] |
| **Empirical p1–p99** | ≈ 0.005 – 0.99 |
| **High value** | Wallet appears on both sides of trades |
| **Low value** | No self-matching detected |
| **Mutable** | ✅ |

Rate at which the wallet appears as both initiator and responder. Kept as a
separate column for ensemble diversity; future implementations may diverge from
`round_trip_frequency` when order-book fill matching data is available.

---

### 2.5 · `order_cancellation_rate`

| Attribute | Value |
|-----------|-------|
| **Formula** | cancelled_offers / total_manage_offer_operations |
| **Range** | [0.0, 1.0] |
| **Empirical p1–p99** | ≈ 0.016 – 0.89 |
| **High value** | High proportion of offers cancelled before fill — consistent with layering or spoofing |
| **Low value** | Most offers result in fills or updates |
| **Mutable** | ✅ |

Computed from order-book events loaded by
`ingestion.orderbook_loader.load_accounts_orderbook_events`. Defaults to
`0.0` when order-book ingestion is skipped (`--no-orderbook` flag).

## 4 · Wallet Graph Features

These features describe the wallet's position and behaviour within the on-chain
funding graph built from `AccountActivity` records.

### 4.1 · `funding_source_similarity`

| Attribute | Value |
|-----------|-------|
| **Formula** | Cosine similarity of multi-hop funding fingerprint vs. known wash-trade cluster centroids |
| **Range** | [0.0, 1.0] |
| **Empirical p1–p99** | ≈ 0.006 – 0.94 |
| **High value** | Wallet shares infrastructure (funding path) with previously flagged wash-trade clusters |
| **Low value** | Wallet's funding sources are dissimilar from known wash-trade patterns |
| **Mutable** | ❌ (funding path is set at account creation; rarely changes) |

Computed by `detection.wallet_graph.compute_wallet_graph_metrics`. Uses
multi-hop ancestor traversal (`max_depth=4`) so intermediate hop accounts do
not mask the shared root funder.

---

### 4.2 · `network_centrality`

| Attribute | Value |
|-----------|-------|
| **Formula** | Betweenness centrality in the directed funding graph |
| **Range** | [0.0, 1.0] |
| **Empirical p1–p99** | ≈ 0.013 – 0.94 |
| **High value** | Wallet acts as a routing hub between many other accounts — common in the "controller" accounts of wash-trade rings |
| **Low value** | Wallet is a leaf node or peripheral participant |
| **Mutable** | ❌ |

---

### 4.3 · `account_age_days`

| Attribute | Value |
|-----------|-------|
| **Formula** | (scoring_timestamp − account_created_at).total_seconds() / 86400 |
| **Range** | [0.0, ∞) |
| **Empirical p1–p99** | ≈ 3.4 – 943 days |
| **High value** | Long-established account — slightly lower prior probability of being throwaway wash-trade account |
| **Low value** | Very new account (< 7 days) combined with high risk score is a strong throwaway signal |
| **Mutable** | ✅ (increases monotonically over time) |

Defaults to `0.0` when no `AccountActivity` record is available (`--no-graph` flag).

---

### 4.4 · `in_wash_trading_ring`

| Attribute | Value |
|-----------|-------|
| **Formula** | 1 if Louvain community_id ≠ −1 (ring size ≥ `min_ring_size`), else 0 |
| **Range** | {0, 1} (boolean, stored as int) |
| **Empirical p1–p99** | 0 or 1 |
| **High value (1)** | Wallet belongs to a detected wash-trading ring — ensemble weight elevated |
| **Low value (0)** | Not assigned to a ring by community detection |
| **Mutable** | ❌ (ring membership is determined at batch-scoring time from the funding graph snapshot) |

Only present when a funding graph is supplied. Absent for graph-less scoring flows.

---

### 4.5 · `ring_size`

| Attribute | Value |
|-----------|-------|
| **Formula** | Number of wallets in the Louvain community this wallet belongs to |
| **Range** | [0, n_wallets_in_graph] |
| **Empirical p1–p99** | 0 – varies |
| **High value** | Large coordinated ring — more severe manipulation signal |
| **Low value** | 0 = not in a ring; 1–2 = below min_ring_size, treated as no ring |
| **Mutable** | ❌ |

---

### 4.6 · `ring_internal_density`

| Attribute | Value |
|-----------|-------|
| **Formula** | actual_intra_ring_edges / possible_intra_ring_edges |
| **Range** | [0.0, 1.0] |
| **Empirical p1–p99** | 0.0 – varies |
| **High value** | Near 1.0 — ring is nearly fully connected (clique); highly organised wash-trade network |
| **Low value** | 0.0 — not in a ring or sparsely connected ring |
| **Mutable** | ❌ |

## 5 · Cross-Asset Coordination Features

These six features detect coordinated activity *across* multiple asset pairs.
They require `all_pairs_df` to be supplied to `build_feature_matrix`; when
omitted all six default to `0.0`.

### 5.1 · `cross_pair_trade_synchrony`

| Attribute | Value |
|-----------|-------|
| **Formula** | trades_with_simultaneous_other_pair_activity(window=30 s) / total_trades |
| **Range** | [0.0, 1.0] |
| **Empirical p1–p99** | ≈ 0.016 – 0.97 |
| **High value** | Wallet regularly trades on multiple pairs within 30 seconds — mechanically coordinated |
| **Low value** | Pair activity is independent |
| **Mutable** | ✅ |

Window size is configured by `config.CROSS_PAIR_SYNCHRONY_WINDOW_SECONDS` (default 30 s).

---

### 5.2 · `net_asset_flow_deviation`

| Attribute | Value |
|-----------|-------|
| **Formula** | max(|net_flow_per_asset|) / total_volume |
| **Range** | [0.0, 1.0+] — values above 1.0 are theoretically possible due to multi-asset accounting |
| **Empirical p1–p99** | ≈ 0.012 – 0.97 |
| **High value** | Near 1.0 — wallet has large net flows; genuine inventory movement |
| **Low value** | Near 0.0 — wallet ends up with roughly the same assets it started with; closed-cycle wash trade |
| **Mutable** | ✅ |

The defining characteristic of wash trading is that no real economic transfer
occurs. A net deviation near zero across all assets is the clearest signal.

---

### 5.3 · `cross_pair_counterparty_overlap`

| Attribute | Value |
|-----------|-------|
| **Formula** | Jaccard(counterparties_pair_A, counterparties_pair_B) for the two busiest pairs |
| **Range** | [0.0, 1.0] |
| **Empirical p1–p99** | ≈ 0.010 – 0.98 |
| **High value** | Wallet reuses the same sock-puppet counterparties across pairs — coordinated network |
| **Low value** | Counterparty sets are pair-specific; consistent with legitimate market making |
| **Mutable** | ✅ |

---

### 5.4 · `cross_pair_volume_correlation`

| Attribute | Value |
|-----------|-------|
| **Formula** | Pearson correlation of per-minute volumes between the two busiest pairs |
| **Range** | [−1.0, 1.0] |
| **Empirical p1–p99** | ≈ −0.41 – 0.99 |
| **High value** | Near +1.0 — volumes spike simultaneously across pairs; mechanically coordinated |
| **Low value** | Near −1.0 or 0.0 — pair volumes are independent |
| **Mutable** | ✅ |

---

### 5.5 · `pair_diversity_score`

| Attribute | Value |
|-----------|-------|
| **Formula** | H(pair_volume_distribution) / log₂(n_pairs) where H is Shannon entropy |
| **Range** | [0.0, 1.0] |
| **Empirical p1–p99** | ≈ 0.031 – 0.95 |
| **High value** | Near 1.0 — volume evenly spread across many pairs; may indicate synthetic diversification |
| **Low value** | Near 0.0 — heavily concentrated on one pair |
| **Mutable** | ✅ |

---

### 5.6 · `cross_pair_mad_std`

| Attribute | Value |
|-----------|-------|
| **Formula** | std(benford_mad per pair, averaged across windows) |
| **Range** | [0.0, ~0.08] |
| **Empirical p1–p99** | ≈ 0.001 – 0.050 |
| **High value** | Benford non-conformity varies widely between pairs — only some pairs manipulated |
| **Low value** | Near 0.0 — Benford conformity is equally bad (or good) on every pair; systematic bot pattern |
| **Mutable** | ✅ |

## 6 · Hardening Features

These three features are designed to resist common adversarial evasion
strategies (temporal spreading, amount rounding, rotating counterparties).

### 6.1 · `inter_arrival_cv`

| Attribute | Value |
|-----------|-------|
| **Formula** | std(inter_trade_intervals) / mean(inter_trade_intervals) |
| **Range** | [0.0, ∞) |
| **Empirical p1–p99** | ≈ 0.005 – 1.82 |
| **High value** | Irregular trade timing — human-like or organic |
| **Low value** | Near 0.0 — trades arrive at perfectly regular intervals; bot with TemporalSpreading evasion |
| **Mutable** | ✅ |

A low CV is the fingerprint of a bot that spaces trades exactly N seconds apart
to evade burst-detection features.

---

### 6.2 · `entropy_of_amounts`

| Attribute | Value |
|-----------|-------|
| **Formula** | Shannon entropy of the amount histogram (50 bins, log₂) |
| **Range** | [0.0, log₂(50) ≈ 5.64] |
| **Empirical p1–p99** | ≈ 0.05 – 5.94 |
| **High value** | Amounts are broadly distributed — diverse, organic trading |
| **Low value** | Near 0.0 — amounts cluster on a handful of values; bot using fixed lot sizes |
| **Mutable** | ✅ |

Hardened against AmountRounding attacks. A bot that rounds amounts to 2
significant figures collapses entropy below ~1.0 bit even when it passes
the Benford chi-square test in log space.

---

### 6.3 · `cross_wallet_volume_corr`

| Attribute | Value |
|-----------|-------|
| **Formula** | Pearson correlation of per-minute volumes between the two most frequent counterparties (lag 0) |
| **Range** | [−1.0, 1.0] |
| **Empirical p1–p99** | ≈ −0.29 – 1.00 |
| **High value** | Near +1.0 — both main counterparties spike volume simultaneously; coordinated ring |
| **Low value** | Near −1.0 or 0.0 — counterparty volumes are independent |
| **Mutable** | ✅ |

## 8 · GNN Embedding Features

### 8.1 · `gnn_0` … `gnn_31`

| Attribute | Value |
|-----------|-------|
| **Formula** | GraphSAGE embedding of the wallet node in the combined funding + co-trade graph |
| **Range** | (−∞, ∞) — real-valued; empirically bounded to roughly [−2.0, 2.0] after training |
| **Empirical p1–p99** | 0.0 (all zeros before first training run) |
| **High / Low value** | Dimensions have no individual human-readable interpretation; the vector as a whole encodes multi-hop structural position in the wash-trade network |

training time. See [`docs/gnn_architecture.md`](../docs/gnn_architecture.md) for
the full architecture and training procedure.

| **Formula** | std(inter_trade_intervals) / mean(inter_trade_intervals) (coefficient of variation) |
| **Range** | [0.0, ∞) |
| **High value** | Highly irregular — human-like |
| **Low value** | Near 0.0 — mechanical regularity; bot |
| **Mutable** | ✅ |

Note: overlaps with `inter_arrival_cv` but computed from a different data source
(Horizon operation history vs. trade DataFrame). The two features may diverge
after adversarial perturbation.

---

### 9.5 · `bot_op_entropy`

| Attribute | Value |
|-----------|-------|
| **Formula** | Shannon entropy (bits) of the distribution of Horizon operation types |
| **Range** | [0.0, log₂(n_op_types)] — typically [0.0, 5.0] |
| **High value** | Broad operation diversity — human-like on-chain footprint |
| **Low value** | Near 0.0 — wallet only performs one or two operation types; bot fingerprint |
| **Mutable** | ✅ |

### 9.6 · `bridge_round_trip_ratio`

| Attribute | Value |
|-----------|-------|
| **Formula** | Bridge-in volume that returns via a bridge-out within N ledgers / total bridge volume |
| **Range** | [0.0, 1.0] |
| **High value** | Assets bridged in and immediately bridged out — round-trip wash trade using a cross-chain bridge |
| **Low value** | No bridge round-trips detected |
| **Mutable** | ✅ |

Populated by callers that supply bridge transaction data. Defaults to `0.0`
in the standard pipeline.
